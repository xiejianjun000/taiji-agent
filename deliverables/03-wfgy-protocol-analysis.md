## 1. 核心模块分析

太极 Verify 协议的核心由五大力学模块组成，对应太极五卦：坤守(☷)、乾进(☰)、复归(☳)、巽调(☴)、阴阳距(ΔS)。每个模块有明确的数学定义和工程接口。

### 1.1 阴阳距 (ΔS / Yin-Yang Distance)

#### 数学定义

ΔS 是衡量当前推理状态 I 与目标状态 G 之间语义张力的标量度量：

```
ΔS = 1 - cos(I, G)
```

其中:
- **I** (Input/当前状态): 当前候选回答或推理链的语义嵌入向量
- **G** (Goal/目标状态): 目标语义嵌入向量，由用户请求、系统规则和可信上下文联合推导

**锚点扩展版本**（当存在命名实体标记时）:

```
sim_est = w_e · sim(entities) + w_r · sim(relations) + w_c · sim(constraints)
ΔS = 1 - sim_est
```

默认权重: `w_e = 0.5, w_r = 0.3, w_c = 0.2`

性质:
- ΔS ∈ [0, 1], 0 表示语义完全对齐，1 表示正交（无关）
- 当锚点存在时，sim_est 需归一化至 [0, 1]

#### 阈值体系

| 区域 | 阈值范围 | 颜色编码 | 含义 | 触发动作 |
|------|----------|----------|------|----------|
| 安全 (Safe) | ΔS < 0.40 | 🟢 | 语义对齐良好，可继续 | 正常推理 |
| 过渡 (Transit) | 0.40 ≤ ΔS < 0.60 | 🟡 | 存在语义漂移迹象 | 检查 λ 观察值 |
| 风险 (Risk) | 0.60 ≤ ΔS < 0.85 | 🟠 | 语义张力过大 | 强制记录内存节点 |
| 危险 (Danger) | ΔS ≥ 0.85 | 🔴 | 语义崩溃风险 | 触发 BBCR 回退 |

#### 政务场景应用

在政务场景中，阴阳距可用于:
- **政策问答一致性检查**: 计算模型回答 I 与官方政策文档 G 的语义距离，确保回答不偏离政策口径
- **多轮对话漂移检测**: 监测咨询对话中语义是否从原始问题渐变偏离
- **跨部门知识对齐**: 评估不同部门对同一政策的理解偏差
- **舆情分析张力测量**: 测量公众言论与官方表述的语义偏离程度

### 1.2 坤守 (BBMC / Kun Guard — 语义残差修正)

#### 数学定义

坤守模块的核心是语义残差最小化:

```
B = I - G + m · c²
```

其中:
- **B**: 语义残差向量，需最小化的 ‖B‖₂
- **I**: 当前语义嵌入向量
- **G**: 基准真值嵌入向量
- **m**: 匹配系数，取值范围 [0, 1]，默认 cos(I, G)
- **c**: 上下文因子，计算方式 `scaled Δtoken / 100`，限制在 [0.2, 1.5]

从 Flagship 协议中的推导关系:
- B_s = ΔS = 1 - cos(I, G) [标量版本]
- BBMC 负责将残差 B 最小化至可接受范围

#### 残差阈值

| 参数 | 默认值 | 含义 | 触发条件 |
|------|--------|------|----------|
| B_c (崩溃阈值) | 0.85 | 残差的范数上限 | ‖B‖ ≥ B_c 触发复归 |
| 记录阈值 | 0.60 | 强制记录语义节点 | ΔS > 0.60 |
| 典范阈值 | 0.35 | 记录为优秀样例 | ΔS < 0.35 |

#### 与知识库向量检索集成方式

坤守模块与知识库的集成遵循以下管道:

1. **检索阶段**: 使用传统向量检索获取候选文档集 D = {d₁, d₂, ..., dₖ}
2. **残差计算**: 对每个候选 dᵢ，计算 Bᵢ = I - emb(dᵢ) + m·c²
3. **残差排序**: 按 ‖Bᵢ‖₂ 升序排列，取残差最小的前 k' 个
4. **融合策略**:
   - 若 min(‖Bᵢ‖₂) < 0.40: 直接采纳该文档
   - 若 0.40 ≤ min(‖Bᵢ‖₂) < 0.60: 混合检索结果与当前推理
   - 若 min(‖Bᵢ‖₂) ≥ 0.60: 触发乾进多路径探索

### 1.3 乾进 (BBPF / Qian Advance — 语义演进建模)

#### 数学定义

乾进模块实现多路径语义演进:

```
x_next = x + Σᵢ Vᵢ(εᵢ, C) + Σⱼ Wⱼ(dt, dO) · Pⱼ
```

其中:
- **x**: 当前语义状态向量
- **Vᵢ**: 第 i 条路径的扰动函数，基于创意意图 top-k
- **εᵢ**: 第 i 条路径的噪声尺度
- **C**: 上下文约束
- **Wⱼ**: 第 j 条路径的动态权重，Wⱼ = exp(-ΔSⱼ)
- **Pⱼ**: 第 j 条路径的优先概率
- **N**: 扰动路径数
- **M**: 加权路径数

#### 稳定性得分

```
f_S = 1 / (1 + mean(Δ))
```

其中 Δ 是所有候选路径与目标的阴阳距向量。f_S ∈ (0, 1]:
- f_S → 1.0: 所有路径高度对齐，状态稳定
- f_S → 0.0: 所有路径严重漂移，需触发复归

耦合器判断（从 Flagship）:
```
prog = max(zeta_min, ΔS_prev - ΔS_now)
P = pow(prog, omega)
W_c = clip(B_s · P + Phi, -θ_c, +θ_c)
```

乾进桥接（Bridge）条件:
- `ΔS decreases` 且 `W_c < 0.5 · θ_c` 且 WDT 检查通过

#### k_paths 和 noise_scale 参数

| 参数 | 默认值 | 含义 | 取值范围 |
|------|--------|------|----------|
| k_paths | 3-5 (建议) | 并行探索路径数 | [1, 10] |
| noise_scale | 0.1 (建议) | 扰动噪声幅度 | [0.01, 0.5] |
| θ_c (耦合阈值) | 0.75 | 耦合器输出裁剪上限 | [0.5, 1.0] |
| ζ_min (最小推进) | 0.10 | 最小推进步长 | [0.01, 0.30] |

### 1.4 复归 (BBCR / Fu Return — 崩溃逆转恢复)

#### 状态机定义

复归模块实现三段式崩溃逆转:

```
if (norm(B) ≥ B_c) or (f(S) < ε):
    → collpase()       # 阶段 1: 崩溃检测
    → reset_state(S, dB)  # 阶段 2: 状态重置
    → rebirth(S_next, dB) # 阶段 3: 重生恢复
```

**状态机流转**:

```
┌─────────────┐     norm(B) ≥ B_c     ┌─────────────┐
│  正常推理态  │ ────────────────────→ │  崩溃检测态  │
│  (Normal)   │    或 f(S) < ε        │  (Collapse) │
└─────────────┘                       └─────────────┘
       ↑                                     │
       │                                     ▼
       │                              ┌─────────────┐
       │     重生成功                   │  状态重置态   │
       │ ←─────────────────────────── │  (Reset)     │
       │                              └──────┬──────┘
       │                                     │
       │                                     ▼
       │                              ┌─────────────┐
       │     重生失败                   │  重生恢复态   │
       └────────────────────────────── │  (Rebirth)  │
          (请求用户确认)               └─────────────┘
```

#### 李雅普诺夫指数 λ 计算

λ (lambda_observe) 描述语义系统的动态稳定性:

```
Delta = ΔS_t - ΔS_{t-1}
E_resonance = rolling_mean(ΔS, window=min(t, 5))
```

| λ 状态 | 检测条件 | 含义 | 动作 |
|--------|----------|------|------|
| convergent (收敛) → | Delta ≤ -0.02 且 E_resonance 非增 | 系统趋向稳定 | 继续推进 |
| recursive (递归) <> | \|Delta\| < 0.02 且 E_resonance 平坦 | 系统在局部振荡 | 触发乾进探索 |
| divergent (发散) ← | Delta ∈ (-0.02, +0.04] 且震荡 | 系统开始漂移 | 加强巽调调制 |
| chaotic (混沌) × | Delta > +0.04 或锚点冲突 | 系统失稳 | 立即触发复归 |

#### B_c 和 ε 阈值调整

| 参数 | 默认值 | 调整方向 | 效果 |
|------|--------|----------|------|
| B_c (崩溃阈值) | 0.85 | ↑ 增大 | 减少崩溃触发，允许更大语义跳跃 |
| B_c (崩溃阈值) | 0.85 | ↓ 减小 | 增加崩溃触发，更严格语义控制 |
| ε (稳定性阈值) | 0.0 | ↑ 增大 | 降低 f(S) 要求，更容易触发复归 |
| ε (稳定性阈值) | 0.0 | ↓ 减小 | 提高 f(S) 要求，减少复归触发 |

### 1.5 巽调 (BBAM / Xun Tune — 注意力调节)

#### 数学定义

巽调模块通过方差门控实现注意力重平衡:

```
âᵢ = aᵢ · exp(-γ · σ(a))
```

其中:
- **aᵢ**: 原始注意力分数
- **σ(a)**: 注意力分数分布的标准差
- **γ**: 调制因子
- **âᵢ**: 调制后的注意力分数

**α_blend 混合机制** (来自 Flagship):
```
α_blend = clip(0.50 + k_c · tanh(W_c), 0.35, 0.65)
最终注意力 = blend(α_blend, a_ref)  // a_ref = uniform_attention
```

#### γ 参数分析

| γ 值 | 效果 | 场景 |
|------|------|------|
| γ = 0 | 无调制，等同标准注意力 | 基线模式 |
| γ = 0.618 (默认) | 太极黄金比例，平衡探索-利用 | 通用场景 |
| γ < 0.618 | 弱调制，保留注意力多样性 | 创意发散任务 |
| γ > 0.618 | 强调制，聚焦关键信号 | 精确问答、政策检索 |

tanh 映射特性:
- k_c = 0.25 (默认)，确保 α_blend ∈ [0.35, 0.65]
- 当 W_c 为负（推进不足），α_blend 偏向 0.35（更均匀）
- 当 W_c 为正（推进良好），α_blend 偏向 0.65（更聚焦）

## 2. 北辰编译器分析

### 2.1 执行管道 6 步骤

北辰编译器将用户自然语言请求转化为可验证的结构化执行：

```
Step 1: GOAL_COMPILATION (目标编译)
    ↓
Step 2: TASK_GRAPH (任务图构建)
    ↓
Step 3: COUNT_BOARD [注：与 ATOM_TABLE 合并为一步]
    ↓
Step 4: ATOM_TABLE (原子任务表)
    ↓
Step 5: EXECUTION_TOKEN_BOARD (执行令牌板)
    ↓
Step 6: ROUND_LOCK (回合锁定)
    ↓
[Round Body: 执行锁定原子]
    ↓
Step 7: DOWNSTREAM_LEAK_AUDIT (下游泄漏审计)
    ↓
Step 8: ROUND_RESULT (回合记录)
    ↓
[Package Records]
    ↓
Step 9: CLOSURE_RECORD (闭包记录)
```

**核心6步骤精简版**：

| 步骤 | 对象 | 目的 | 关键字段 |
|------|------|------|----------|
| 1 | GOAL_COMPILATION | 用户语言→可执行结构 | Goal_ID, Goal_Class, Status, Mixed_Goal_Flag |
| 2 | TASK_GRAPH | 原子映射+依赖+阻塞 | Atom_Nodes, Dependency_Edges, Blocking_Edges |
| 3 | ATOM_TABLE | 记录每个原子的规范模式 | Atom_ID, Class, State, Blocked_By, Success_Test |
| 4 | EXECUTION_TOKEN_BOARD | 判定哪个原子可执行 | Token_Status (GRANTED/DENIED/PENDING), Denial_Class |
| 5 | ROUND_LOCK | 冻结当前回合唯一活性原子 | Active_Atom_ID, Only_Do, Legal_Output_Kind |
| 6 | CLOSURE_RECORD | 验证和闭包记录 | Verified, Missing, Residual_ΔS, Result |

### 2.2 Task Atom 结构

#### 原子规范模式

```
ATOM_RECORD:
  - Atom_ID:          唯一标识符
  - Atom_Label:       可读标签
  - Atom_Class:       原子类别 [8种法定类别]
  - Target_Object:    目标对象描述
  - Truth_Bearing_Flag:  是否为真值承载
  - Expression_Flag:      是否为表达工作
  - Closure_Critical_Flag: 是否为闭包关键
  - Activation_Flag:      是否为激活原子
  - Legality_Floor_Flag:  是否为合法性下限
  - Claim_Ceiling_Flag:   是否为声明上限
  - Legal_Output_Kind:    合法输出类型
  - Blocked_By:           被哪些原子阻塞
  - May_Start_Only_If:    启动前置条件
  - Success_Test:         成功判定条件
  - Failure_Test:         失败判定条件
  - Claim_Ceiling:        声明上限描述
  - Current_State:        当前状态
  - Downstream_Atoms:     下游原子列表
```

#### 8种法定原子类别

| 类别 | 含义 | 真值承载 | 示例 |
|------|------|----------|------|
| DEFINE | 定义目标对象/真值对象/边界 | 是 | "定义密封目标" |
| LOCATE | 定位缺口/阻塞/依赖/证据 | 是 | "定位当前缺口" |
| DERIVE | 计算/推理/推导公式 | 是 | "推导缺失数学" |
| PATCH | 修改/修复/形式化/绑定 | 是 | "修补参数A形式化" |
| VERIFY | 测试/攻击/验证/证伪 | 是 | "验证密封结果" |
| DECIDE | 产生通过/不通过判断 | 是 | "判断密封通过" |
| ORGANIZE | 列出/排序/表格化已验证事实 | 否 | "组织已验证事实" |
| WRITE | 散文/README/公开说明 | 否 | "从事实撰写文章" |

#### Active/Blocked 状态转换

```
                         ┌──────────────┐
                         │  NOT_STARTED │
                         │  (未开始)     │
                         └──────┬───────┘
                                │
                   ┌────────────┼────────────┐
                   ▼            ▼            ▼
            ┌───────────┐ ┌──────────┐ ┌──────────┐
            │  BLOCKED  │ │  ACTIVE  │ │ DEFERRED │
            │ (阻塞)    │ │ (活性)   │ │ (推迟)   │
            └─────┬─────┘ └────┬─────┘ └──────────┘
                  │            │
                  │            ▼
                  │     ┌──────────────┐
                  │     │ LOCALLY_DONE │
                  │     │ (本地完成)   │
                  │     └──────┬───────┘
                  │            │
                  │            ▼
                  │     ┌──────────────┐
                  │     │  VERIFIED    │
                  │     │ (已验证)     │
                  │     └──────┬───────┘
                  │            │
                  ▼            ▼
            ┌──────────┐ ┌──────────┐
            │ REJECTED │ │(解锁下游)│
            │ (拒绝)   │ │          │
            └──────────┘ └──────────┘
```

**阻塞转换规则**:
- 只有当上游原子的 Current_State = VERIFIED 时，下游 BLOCKED 原子才可解锁为 ACTIVE
- LOCALLY_DONE, PROBABLY_DONE, LOOKS_COMPLETE 为非法解锁状态
- Active_Atom_Count 在任意执行回合必须等于 1

### 2.3 下游泄漏审计机制

下游泄漏审计(DOWNSTREAM_LEAK_AUDIT)是北辰编译器的关键校验步骤：

```
DOWNSTREAM_LEAK_AUDIT:
  - Active_Atom_ID:       当前执行原子
  - Expected_Output_Kind: 期望的输出类型
  - Leak_Detected:        是否检测到泄漏 (0/1)
  - Leak_Source_Class:    泄漏来源原子类别
  - Leak_Fragment_Count:  泄漏片段计数
  - Leak_Disposition:     处理方式
```

**典型泄漏模式**:

| 当前回合 | 非法泄漏内容 | 检测方式 |
|----------|-------------|----------|
| PATCH | 散文解释 | 输出包含非技术性自然语言 |
| VERIFY | README 语言 | 输出包含宣传/说明性内容 |
| DERIVE | 公开叙述 | 输出包含故事性语言 |
| LOCATE | 摘要散文 | 输出包含不必要的概述 |

**泄漏处理规则**:
- Leak_Detected = 1 → 回合失败（除非显式授权草稿壳模式）
- 预激活声明泄漏: 当活跃回合为 DEFINE/LOCATE/DERIVE/PATCH/VERIFY 且输出包含非必需的声明上限散文时，同样判定泄漏
- 修复: 回退至仅产生合法的输出类型，去除所有非法内容

## 3. 失败模式分析

### 3.1 16 种失败模式完整列表

基于 Problem Map 1.0 的 16 种可再现失败模式:

| 编号 | 名称 | 层级 | 检测条件 | 修复动作 | 难度 |
|------|------|------|----------|----------|------|
| 1 | **幻觉与分块漂移** [IN] | 输入与检索 | ΔS(问题, 检索内容) > 0.45 | 应用坤守残差修正，缩小检索-语义鸿沟 | 中 |
| 2 | **解释崩塌** [RE] | 推理与规划 | 分块正确但逻辑错误，λ = divergent | 触发复归找回，重新锚定语义节点 | 高 |
| 3 | **长推理链漂移** [RE] | 推理与规划 | 多步任务中 ΔS 逐步增大且 λ 转为 chaotic | 每步插入阴阳距探测点，必要时分支乾进 | 高 |
| 4 | **虚张声势/过度自信** [RE] | 推理与规划 | 自信但无依据的回答，ΔS 低但覆盖率低 | 应用知识边界探测，强制 BBCR 边界检查 | 高 |
| 5 | **语义≠嵌入** [IN] | 输入与检索 | cosine 匹配高但语义不相关 | 引入锚点扩展 sim_est，降低纯 cosine 权重 | 中 |
| 6 | **逻辑崩溃与恢复** [RE] | 推理与规划 | norm(B) ≥ B_c 或 f(S) < ε | 执行三段式复归: collapse→reset→rebirth | 很高 |
| 7 | **跨会话记忆断裂** [ST] | 状态与上下文 | 新会话无法回忆之前内容 | 语义树持久化 + 节点检索 | 高 |
| 8 | **调试黑盒** [IN] | 输入与检索 | 无故障路径可见性 | 启用阴阳距日志 + λ 观察值跟踪 | 中 |
| 9 | **熵崩溃** [ST] | 状态与上下文 | 注意力熔化，输出不连贯，σ(a) 异常 | 触发巽调方差门控，降低 γ 值 | 高 |
| 10 | **创意冻结** [RE] | 推理与规划 | 输出平直、字面，k_paths 探索不足 | 增加乾进 k_paths 和 noise_scale | 中 |
| 11 | **符号崩塌** [RE] | 推理与规划 | 抽象/逻辑提示失效，λ = chaotic | 使用锚点约束 + 复归符号重建 | 很高 |

### 4.3 太极 Verify 的技术定位

太极 Verify 协议的技术定位是 **"语义防火墙"** ——在 LLM 生成之前检查语义场的稳定性，而非在生成之后修补错误。其核心创新在于:

1. **量化的语义张力度量**: 阴阳距 ΔS 提供了可计算的语义对齐指标
2. **多层级的力学调控**: 坤守(稳定)→乾进(探索)→复归(恢复)→巽调(聚焦)形成闭环
3. **执行的原子化约束**: 北辰编译器确保每次只做一件合法工作
4. **失败模式的结构化修补**: 16 种失败模式均有可重复的诊断和修复路径
