## 二、运行时层：Harness 整合分析

### 2.1 EventBus 事件中枢

**现状**：Harness TS版有37个事件（10个可修改），OpenTaiji Python版有基础实现

| 对比维度 | Harness (TypeScript) | OpenTaiji (Python) |
|----------|---------------------|-------------------|
| 事件数量 | 37个，分类清晰 | 基础实现 |
| 类型安全 | 编译时类型推导 | 无类型约束 |
| 可修改事件 | 白名单控制 | 所有handler可中断 |
| 历史记录 | 无内置 | 1000条内存记录 |
| 同步发射 | 无 | `emit_sync()` 支持 |

**能力评估**：✅ **可直接复用，需扩展**

**扩展需求**：
1. 增加 `gov:*` 命名空间（政务特有事件）
2. 增加事件溯源（Event Sourcing）持久化
3. 增加异步中间件支持
4. 增加分布式支持（Redis/NATS桥接）

### 2.2 Plugin System

**现状**：Harness Plugin接口已定义，Python版缺失

```typescript
interface HarnessPlugin {
  id: string; name: string; version: string;
  activate(ctx: PluginContext): Promise<void>;
  deactivate(): Promise<void>;
  tools?: ToolDefinition[];
  providers?: LLMProvider[];
  hooks?: AnyHookRegistration[];
}
```

**能力评估**：⚠️ **需Python版从零构建**

**设计要点**：
- Taiji Verify Plugin：拦截 `tool:request`、`prompt:assemble`、`llm:response` 做验证
- GovMCP Plugin：提供政务协议适配、MCP能力声明
- 生命周期：loading/error/degraded 状态、热重载、依赖解析

### 2.3 Agent Loop 整合

**现状**：Harness Loop使用ReAct模式，Provider接口已抽象

```
for iteration = 1 to maxIterations:
  1. Emit loop:iteration_start (可中止)
  2. assemblePrompt(system/soul/skills/messages/tools)
  3. Emit llm:request (可中止)
  4. provider.chat(request) → AsyncGenerator<ChatChunk>
  5. Emit llm:response
  6. if no toolCalls → break
  7. for each toolCall: emit tool:request → execute
```

**能力评估**：⚠️ **需替换Provider为Hermes**

**整合路径**：
1. Hermes作为Python侧的LLM统一入口
2. 通过gRPC桥接暴露给Harness
3. 或直接在Python侧构建完整Agent Loop（短期推荐）

### 2.4 HITL 人工介入

**现状**：Harness FeedbackManager实现完整审批流程，GovMCP有ApprovalFlow引擎

| 组件 | 能力 | 状态 |
|------|------|------|
| Harness FeedbackManager | 14.4KB实现，多类型确认 | ✅ |
| GovMCP ApprovalFlow | 多级审批链、状态机、超时处理 | ✅ |
| 审批→Harness回调 | 缺失 | ⚠️ 需开发 |

**能力评估**：⚠️ **需对接两个审批系统**

**对接方案**：
- Webhook触发审批：Harness Approval Step → GovMCP `initiate_approval_workflow`
- 状态轮询：`query_approval_progress` → 回调Harness Webhook
- 政务特有：多级审批链（科员→科长→处长→局长）

### 2.5 Docker Sandbox

**现状**：Harness有完整Docker沙箱实现（热/冷容器、资源限制、超时控制）

**能力评估**：⚠️ **需Python重新实现**

## 四、差异化灵魂层：Taiji Verify 整合分析

### 4.1 阴阳距（ΔS）数学定义与阈值体系

**数学定义**：
```
ΔS = 1 - cos(I, G)
```
- I：当前语义嵌入向量
- G：目标语义嵌入向量

**锚点扩展**（含命名实体时）：
```
sim_est = w_e·sim(entities) + w_r·sim(relations) + w_c·sim(constraints)
ΔS = 1 - sim_est
```

**阈值体系**：

| 区域 | 阈值 | 颜色 | 触发动作 |
|------|------|------|----------|
| 安全 (Safe) | ΔS < 0.40 | 🟢 | 正常推理 |
| 过渡 (Transit) | 0.40 ≤ ΔS < 0.60 | 🟡 | 检查λ观察值 |
| 风险 (Risk) | 0.60 ≤ ΔS < 0.85 | 🟠 | 强制记录内存节点 |
| 危险 (Danger) | ΔS ≥ 0.85 | 🔴 | 触发复归(Fu Return)回退 |

**能力评估**：✅ **完整公式，可立即实现**

### 4.2 四大力学模块

| 模块 | 数学定义 | 工程化难度 | 优先级 |
|------|----------|:----------:|:------:|
| **坤守 (Kun Guard)** | `B = I - G + m·c²` | 中 | P1 |
| **乾进 (Qian Advance)** | `x_next = x + ΣVᵢ + ΣWⱼ·Pⱼ` | 高 | P3 |
| **复归 (Fu Return)** | 三段式：collapse→reset→rebirth | 高 | P2 |
| **巽调 (Xun Tune)** | `âᵢ = aᵢ · exp(-γ·σ(a))` | 低 | P1 |

**坤守**：语义残差最小化，与向量数据库检索集成
**乾进**：多路径语义演进，需实现扰动函数Vᵢ
**复归**：崩溃逆转恢复，基于李雅普诺夫指数λ判定
**巽调**：方差门控注意力重平衡，γ=0.618为黄金比例

### 4.3 北辰编译器（6步执行管道）

```
Step 1: GOAL_COMPILATION    → 用户语言→可执行结构
Step 2: TASK_GRAPH          → 原子映射+依赖+阻塞
Step 3: ATOM_TABLE          → 记录原子规范模式
Step 4: EXECUTION_TOKEN     → 判定可执行原子
Step 5: ROUND_LOCK          → 冻结当前回合唯一活性原子
Step 6: CLOSURE_RECORD      → 验证和闭包记录
```

**8种法定原子类别**：DEFINE / LOCATE / DERIVE / PATCH / VERIFY / DECIDE / ORGANIZE / WRITE

**能力评估**：✅ **规则驱动，可实现**
